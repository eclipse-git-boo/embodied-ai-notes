# Qwen-VLA：统一操作、导航与具身理解的 DiT 动作专家

> 阅读定位：把它看成一条“**强多模态主干 + 连续动作生成器 + 具身提示词**”路线。它的目标不是只把一个机械臂任务做高，而是让同一个模型跨操作、视觉语言导航（VLN）和具身问答迁移。

## 1. 模型原理总览：输入、输出与模块分工

### 1.1 一句话机制

把“图像、自然语言和**本体说明**”送入视觉语言主干；主干提供条件特征，DiT 动作专家从噪声逐步生成一个连续 action chunk。相同的接口也可把导航路径看作动作序列，把视觉语言理解任务看作文本输出任务。

### 1.2 输入—输出契约

| 接口 | 内容 | 作用 |
|---|---|---|
| 观测 $o$ | 当前 RGB 图像；不同任务可有多视角或历史观测 | 定位物体、场景和可操作区域 |
| 语言 $x$ | 操作指令、导航目标、问答文本 | 指定目标与约束 |
| 本体提示 $e$ | 机器人平台、臂型、控制频率、动作空间、预测长度等文本描述 | 让同一参数组知道“这次动作的坐标系和节拍” |
| 时间/噪声 | Flow‑Matching 的时间步 $\tau$ 与噪声动作 $a^\tau$ | 决定本轮去噪位置 |
| 输出 | 操作为连续 chunk $\hat a^{0}$；或导航 waypoint；或文本答案 | 交给评测接口/下游控制层，而不是直接替代安全控制 |

### 1.4 模块责任表

| 模块 | 做什么 | 不做什么/易混点 |
|---|---|---|
| Qwen3.5 视觉语言主干 | 编码图像、语言和本体提示；保留 VQA、空间 grounding 等通用理解能力 | 它不直接逐维回归最终控制量 |
| 本体感知 prompt | 把平台、控制频率和动作形状写成条件 | 不是额外给每台机器人训练一套主干权重 |
| DiT 动作专家 | 读取条件特征、动作 token 和时间步，预测连续动作的速度场 | 不是离散 token 的自回归解码器 |
| 统一动作/轨迹张量与 mask | 把操作 chunk、导航 waypoint 放入同一表示；mask 隐去不适用维度 | mask 不能消除物理本体差异，只让损失接口可共用 |
| 四阶段训练 | 先建立动作先验，再跨模态共训、SFT，最后 RL 对齐任务成功 | RL 不是论文默认给所有用户的本地复现实验 |

### 1.5 从输入到输出的 5 步

1. 将图像 $o$、语言 $x$ 与本体提示 $e$ 拼成条件上下文。
2. 用 Qwen3.5 主干得到语义、空间和任务相关隐藏状态。
3. 初始化噪声动作 $a^1$，把 $a^\tau$、时间步 $\tau$ 与主干状态送进 DiT。
4. DiT 预测速度场，按数值积分逐步得到 $\hat a^0$；无效动作维度由 mask 排除。
5. 将 chunk 按该平台的频率、坐标和安全约束交给**外部**执行接口；本笔记只讨论学习与硬件规划，不在本机控制机器人。

## 2. 为什么这样设计

**问题一：跨本体数据的动作形状不同。** 若只把动作向量硬拼，模型很难知道 7 维单臂、双臂或导航 waypoint 的含义。论文以文本化本体提示描述平台和控制约定，并用统一张量/mask 让共享动作专家看到一致训练接口。

**问题二：只用机器人数据会损失视觉语言常识。** 论文将操作轨迹、人类第一视角、合成轨迹、导航、自动驾驶 VQA、空间 grounding 与通用视觉语言数据混合；其中一般视觉语言目标持续提供对象、文字和空间关系的先验。

**问题三：模仿损失不等于闭环成功。** 所以从大规模预训练开始，再用多任务 SFT 对齐具体执行，最后以仿真 rollout 的任务成功信号做 PPO 式 RL。应把最后一步理解为论文的后训练环节，而非已证明的真机安全方案。

## 3. 训练目标：连续动作的 Flow Matching

对真实动作 chunk $a^0$ 与高斯噪声 $a^1$，在中间时间 $\tau$ 构造 $a^\tau$。动作专家学习速度场 $v_\theta$：

$$
\mathcal{L}_{\mathrm{FM}} = \mathbb{E}_{\tau, a^0, a^1}\left[\left\|v_\theta(a^\tau, \tau \mid o,x,e) - (a^1-a^0)\right\|_2^2\right].
$$

这里 $o,x,e$ 分别是观测、语言和本体条件；$\tau$ 是噪声层级。直觉是：不要一次猜完整轨迹，而是学习“从当前噪声状态朝干净动作应走的方向”。推理时从噪声沿该方向积分得到 action chunk。

在 SFT 中，论文还保留视觉语言的 next-token 目标；论文报告其权重为 0.1，而操作和导航的 Flow‑Matching 项权重为 1.0。这样做是为了避免后训练把主干压缩成只会输出动作的特化网络。

## 4. 数据、任务与结果

### 4.1 四阶段训练配方（论文事实）

| 阶段 | 训练信号 | 想学到什么 |
|---|---|---|
| I：Text-to-Action (T2A) | 语言 + 本体提示 → 动作；论文说明此阶段可不输入图像 | 先让动作专家学习动作语法和不同本体的先验 |
| II：Continued Pre-training | 操作、人类第一视角、合成、导航、轨迹、视觉语言混合数据 | 将视觉 grounding 与连续动作连接起来 |
| III：多任务 SFT | 操作、导航与视觉语言样本 | 提升下游指令跟随与控制精度 |
| IV：RL | 仿真 rollout 的任务成功奖励 | 缩小“模仿得像”与“闭环成功”之间差距 |

## 5. 消融：真正值得抄到实验日志里的结论

可迁移的学习结论是：

- T2A 的价值在于先让动作解码器学会“语言—本体—轨迹”的结构；但其数据比例依赖评测环境，不能照搬。
- 全序列预测在该消融中优于 chunk 模式；图像 token 被加入 T2A 时反而伤害结果，符合“该阶段先学动作先验”的设计。
- 在论文的 SFT 指标上，时间步分布和训练时长有明显稳定区，而非训得越久越好。

## 6. 局限与复现边界

1. **统一接口不等于统一动力学。** prompt/mask 描述动作空间，但接触、时延、相机外参和控制器仍会造成跨本体失配。
2. **RL 复现门槛高。** 论文的 on-policy rollout 依赖仿真基准和并行环境；只复现 SFT 的结果不能声称复现完整 Instruct 训练。
3. **视觉泛化不等于安全。** OOD 成功率不覆盖力控、碰撞、急停、人体共处等真实风险；学习机不应连接执行机构。
4. **公开信息边界。** 本笔记仅记录论文和仓库明确披露的信息；对总训练 token、完整 GPU-hours 等未充分披露项不做虚构估算。

## 7. 全量训练、微调与推理：算力/硬件审计

| 场景 | 论文/官方明确披露 | 规划建议（个人推断，非官方最低配置） | 主要风险 |
|---|---|---|---|
| 从零全量预训练 | 论文给出 Qwen3.5‑4B 主干与约 1.15B 动作专家、四阶段配方；未在论文中给出完整 GPU 数、时长或总 token 的可复现实例 | 不建议个人学习机尝试；该量级需多 GPU 分布式训练、快速共享存储和大规模清洗数据流水线 | 资金、数据版权、训练稳定性与仿真/真机差距 |
| SFT / LoRA | 官方仓库是应优先遵循的唯一配置来源；训练显存、batch 和图像数受版本与分辨率影响 | 先以单机 24–48GB 显存做小数据 smoke test；更稳妥的是 48–80GB 显存或多卡，使用 LoRA/QLoRA、梯度累积、checkpointing | 动作归一化、相机/本体 prompt、数据时间对齐错一项就会“能训练但不能用” |
| 推理评测 | 论文使用连续 action chunk；未给出适用于所有本体的消费级显卡下限 | 24GB 起可做量化/小批评测的规划起点；保留 CPU 内存、相机解码与动作后处理余量，先离线回放 | 时延抖动、控制频率不匹配、量化改变动作精度 |

这台机器定位为**学习与离线复现准备**：先验证数据读取、动作维度、归一化、离线指标和仿真；不要在此处执行真实机器人控制。

## 8. 领域 Q&A：VLA / WAM / VLN 阅读时应追问什么？

**Q1：这是 VLA 还是 VLN，为什么放在 VLA 区？** 主要接口是观测/语言到连续动作，因此是 VLA；同时把 waypoint 轨迹作为统一动作表示来覆盖 VLN。阅读导航结果时要单独看 R2R/RxR 的 SR、SPL、nDTW，不能与机械臂成功率混成一个数字。

**Q2：本体 prompt 真正解决了什么？** 它将“动作值应该怎样解释”显式给模型；它不能自动解决标定、动力学、控制器和相机位姿的现实差异。复现时先检查 prompt、动作维度、控制频率和 horizon 是否与数据完全同义。

**Q3：Flow Matching 对部署意味着什么？** 它输出连续 chunk，通常比逐 token 自回归更适合高频动作；但采样步数、chunk 长度和控制循环共同决定端到端时延，必须离线测量而非只看训练 loss。
